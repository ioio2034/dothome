웹 표준 - 기본

01.웹접근성의 이해

1.웹 접근성이란?

  • 정의 : 웹 콘텐츠라는 대상에 접근할 수 있는 능력
  • 접근하기쉽다 = 이용하기 쉽다.
  • 웹 접근성 중요 요소
    1. 누구나, 어떠한 기술 환경에 구애받지 않음
    2. 전문적인 능력 없이 웹 사이트에서 제공하는 모든 정보

2. 웹 접근성의 범위

  • 웹 접근성의 좁은 의미
    • 다양한 사용자 환경에서의 장애인, 보조기기, 보조기술 = 장애인 접근성
  • 비 장애인에 대한 환경 고려, 다양한 기술환경,사용자 환경 고려
  • 현재 '웹 접근성 품질마크'의 기준
    • 장애인과 보조기술에 관계된 지침, 비장애인도 혜택을 받는 형태로 기술
  • 웹 접근성의 적용범위는 넓은의미의 기대만큼 적용 범위가 넓어질 것으로 예상

3.웹 접근성 관련 용어(1)

  • 웹 접근성의 넓은 의미 : 기본적인 정의, 기대효과
  • 웹 접근성 지침
    • "장애인 접근성" 중심으로 현재까지 ActiveX 사용
    • 시각 장애인등 다양한 장애 유형별 사용자가 이를 키보드로 이용가능함
    • 대체 수단을 통하여 음성 지원 등 다양한 방법으로 설치 및 이용할 수 있다면 웹 접근성을 준수한 것으로 인정
    • 웹 접근성을 준수하면 다양한 기술 환경에서 접근이 가능

4.웹 접근성 관련 용어(2)

  • 웹 표준
    • 웹에서 정한 기술 표준으로 w3c와 같은 표준화 기구에서 승인한 개방형 인터넷 표준
    • 웹 기술에 대한 표준 규격을 공통의 약속
    • = '웹 기술 표준'의 약어
    • 웹 표준의 좁은 의미 : 웹 화면을 구성하는 기술에 대한 표준 규격 =웹 표준 아님
    • 웹 표준의 넓은 의미 : 웹 기술 전반에 대한 표준 규격 = 웹 표준
  • 웹 호환성
    • 크로스 브라우징과 동일한 의미
    • 행정기관이 국민을 대상으로함 - 전자 정부 서비스 등의 웹 사이트를 사용자인 국내외에 거주 하고 있는 국민
    • 어떤 브라우저를 사용하든지 웹 사이트에 접속하여 정상적인 서비스를 받을 수 있어야함
  • 오픈 웹
    • 대표적으로 국내 금융 사이트에서 사용되는 active x 방식의 보안기술 종속된 금융시스템
    • 대한민국 전자정부와 온라인 뱅킹 서비스를 대상으로 함
  • 웹 호환성
    1. 약속된 표준에 의해 서로 다른 종류 기기들이 호환되는 표준 방식 호환성
    2. 비표준에 대해 상호 호환이 가능하도록 하는 비표준 방식 호환성

03. 인식의 용이성(1)

1. 대체 텍스트

  • 개념
    • 텍스트가 아닌 콘텐츠들 즉, 이미지, 이모티콘 문자, 영상, 음성 콘텐츠 등에 대하여 이를 텍스트 정보로 대체 해주는 것
    • 텍스트가 아닌 콘텐츠를 대체하여 제공하는 텍스트 정보
  • 필요성
    • 시각장애인에게 중요한 지표
    • 소리로 모든 내용을 이해하는 시각장애인에게 텍스트 정보가 음성으로 변환이 가능
  • 제공방법 기준
    • 기준 : 텍스트가 아닌 콘텐츠를 이용할 경우, 해당 이미지가 제공하는 의미나 용도를 동일하게 인식할 수 있는 적절한 대체텍스트 제공 시 준수한 것으로 인정
    • 제공방법
      1. 텍스트 이미지에 대체 텍스트를 미제공한 경우 : 이미지 태그의 대체 텍스트를 제공해야 합니다.
      2. 불충분한 대체 텍스트를 제공한 경우 : 제목과 더불어 중요 정보를 같이 제공해야 합니다.
      3. 오타로 표기된 경우 : 오타 수정
      4. input type="image"에 대체 텍스트를 미 제공한 경우 : alt속성 값을 이용해서 대체 텍스트를 제공해야 합니다.
      5. 게시판 내용 중 이미지의 대체 텍스트를 미 제공한 경우 : 해당 이미지의 주제,목적 등을 대체 텍스트로 제공해야 합니다.
      6. 블릿 이미지(+의미 없는 이미지) 대한 대체 텍스트를 미 제공한 경우 : alt="" 속성에 빈칸도 없이 제공하면 화면낭독기에서 낭독하지 않고 넘어갑니다.
      7. 의미없는 이미지에 대한 대체 텍스트를 제공한 경우 : alt="" 제공하지 않아도 됩니다.
      8. 여러 개로 분리한 이미지 조각의 대체 텍스트 제공 : 크기가 가장 큰 이미지에 대체 텍스트를 제공하고 나머지의 이미지에는 빈칸으로 제공합니다.
  • → 대체 텍스트는 이미지의 alt속성 값을 이용해서 제공함
  • → 이미지의 내용이 복잡하거나 길어질 경우에는 longdesc 속성을 이용하여 별도의 파일로 제공

2. 멀티미디어 대체수단

  • 멀티미디어 콘텐츠의 개념
    • 주로 음성을 포함하는 영상 또는 음성으로만 제공되는 콘텐츠 등
    • 멀티미디어 콘텐츠의 대체수단은 음성정보를 대신하여 시각적인 정보인 자막이나 대본 또는 수화 등의 대체해주는 것
  • 필요성
    • 청각장애인에게 중요한 지표
    • 영상 또는 음성 콘텐츠에서 제공되는 소리를 시각적 정보인 자막, 대본, 수화를 통해 인식
  • 검사항목 해설
    • 정보의 정확성과 한 화면에서 볼 수 있도록 제공
    • 정보의 차별이 생기지 않도록 주의
    • 파일로 제공되는 동영상과 음성 및 자막도 평가대상에 포함
    • 닫힌 자막(Closed Caption)을 오디오와 동기화시켜 제공
    • 즉시성이 필요한 경우, 실시간 수화를 제공하거나 자막을 제공
  • 기준
    • 멀티미디어 콘텐츠를 동등하게 인식할 수 있도록 제작하기 위해서 자막, 대본 또는 수화를 제공한 경우 준수한 것으로 인정
  • 오류 유형
    1. 영상, 음성 콘텐츠에 자막, 대본, 수화 중 하나 이상의 대체 수단을 제공하지 않은 경우
    2. 내용 전체를 충분히 설명하지 않고, 요약 정보나 제목만 제공하는 경우
    3. 텍스트만 제공하는 영상 콘텐츠에서 동등한 음성 미 제공 시
  • 주의 사항
    1. 파일로 제공되는 영상, 음성 콘텐츠도 평가 대상에 포함
      • 단, 파일로 제공되는 영상, 음성의 자막은 파일로 제공한 경우도 준수한 것으로 인정함
      • 스트리밍형과 파일형으로 동시에 제공하여 다양한 경로로 정보를 얻을 수 있도록 해주는 것이 바람직
    2. 자막으로 인해 수화가 가리지 않도록 하고, 원고는 페이지를 벗어나지 않는 수준으로 제공해야함
      • 자막으로 인해 수화가 가려지는 경우
      • 원고가 페이지를 벗어나는 경우 - 스크롤 제시
    3. 사용자가 등록한 콘텐츠 제공 시, 자막을 등록하도록 유도하는 문구 또는 자막생성 툴을 제공하는 것을 권장

04. 인식의 용이성(2)

1. 색에 무관한 콘텐츠 인식

  • 개념
    • 정보를 구분하는 구별점을 색상으로만 하지 않고, 색을 배제하더라도 인식할 수 있도록 제공하는 것
  • 목적
    • 시력이 약한 사용자에게 색상이 아닌 다른 방식으로 구분할 수 있는 정보 제공
  • 해설
    • 검사항목 3. 명도 차에 대한 색상 구분은 따로 두지 않음
  • 제공 방법
    • 기준 : 색을 배제하여도 해당 콘텐츠를 인식할 수 있는 정보를 제공한 경우 준수한 것으로 인정
  • 오류 유형
    1. 색상만으로 내용을 분별하도록 제공된 콘텐츠(그래프, 차트, 지도 등)
      : 색의 부분은 보조적으로 제공하고 명확한 수치 파악을 위해 보조선 및 범례를 두어 구분할 수 있도록 합니다.
    2. 페이지 네비게이션, 메뉴, 현재 위치 등에 대해 명암, 패턴 등의 변화 없이 색상의 변환만으로 현재의 위치를 표시한 경우
      : 색상뿐만 아니라 체크 표시를 해서 해당 위치를 인식하기에 용이하게 합니다.
    3. 필수 입력 항목을 색으로만 표시한 경우
      : 색상으로 구별한 정보에 체크표시를 해서 색상이 제공되지 않더라도 패턴을 통해 필수 입력 항목을 구분할 수 있게 제공합니다.

2. 명확한 지시사항 제공

  • 개념
    • 지시사항을 전달하는 콘텐츠에 한정해 적용하는 것
    • 무엇을 지시하는지 명확하게 하기 위해 목적이 분명한 지시가 되어야 하는 것
    • 특정 감각에만 의존하지 않고 다른 감각을 통해서도 지시사항을 인식할 수 있도록 정보를 제공하는 것
  • 목적
    • 특정한 감각에 장애가 있는 사용자가 웹 사이트에서 지시된 내용을 제대로 실행할 수 있도록 함
  • 해설
    • 검사항목 4. 링크 텍스트가 아닌 이미지 텍스트로 만든 링크로 링크 텍스트의 내용과 똑같이 대체 텍스트로 제공하여 시각장애인도 이해 가능
  • 제공방법
    • 기준 : 지시사항 정보를 특정 감각에 의존하지 않고 다양한 감각을 통해 용도나 목적을 이해할 수 있도록 제공한 경우
  • 오류 유형
    1. 색, 크기, 모양, 방향 등으로만 정보를 제공한 경우 : 명확한 목적과 용도를 이해할 수 있게 고유 명사로 정보를 제공합니다.
    2. 전달하고자 하는 지시사항을 소리로만 정보를 제공한 경우 : 청각으로 파악할 수 없는 콘텐츠를 시각적으로도 확인할 수 있도록 제공합니다.

3. 텍스트 콘텐츠의 명도 대비

  • 명도 대비의 개념
    • 텍스트 콘텐츠와 배경 간의 명암의 차이
    • 4.5:1이상을 권장
  • 목적
    • 시력이 약한 사용자를 위한 지침으로 최소한의 명도 차이를 둠으로써 정보를 인식할 수 있도록 함
  • 해설
    • 검사항목 5.본문 콘텐츠에 대해서만 적용
    • 검사 적용 예외
      • 단순히 장식 목적으로만 사용한 텍스트
      • 로고 또는 상호과 같은 텍스트 이미지
      • 마우스나 키보드를 활용하여 초점을 받았을 때 색이나 명도 대비가 변화하는 콘텐츠
      • 사용할 수 없음을 표시하기 위하여 명도 대비를 낮춘 회색의 컨트롤이나 입력 서식 등
  • 제공방법
    • 기준 : 웹페이지 콘텐츠에 텍스트, 이미지 텍스트 정보에 대해 폰트의 크기가 4.5:1이상(18pt이상), 굵은 14pt 이상은 3:1이상의 명도 대비를 제공하는 경우
    • 단, 화면 확대가 가능하도록 구현한 텍스트 콘텐츠의 명도 대비는 3:1까지 준수로 인정함
  • 오류 유형
    1. 텍스트의 규격 정보가 있으며, 보통 크기(18pt 미만, 또는 굵은 14pt 미만)의 텍스트가 4.5:1을 만족하지 않은 경우
      : 텍스트와 배경의 명암차이를 4.5:1비율 이상 유지해 텍스트 읽기가 수월해지게 제공합니다.
    2. 텍스트의 규격 정보가 있으며, 텍스트의 크기가 18pt 이상, 또는 굵은 14pt 이상의 텍스트가 3:1을 만족 하지 않은 경우
      : 텍스트와 배경의 명도를 조절하여 3:1의 비율로 조정합니다.
    3. 이미지 텍스트의 경우 폰트 종류와 상관없이 14pt 크기에 해당하는 18.66pt 미만인 경우 4.5:1이상이거나, 18.66pt 이상인 경우 3:1이상을 만족하지 않은 경우(굵은 14pt 기준은 적용x)
  • 주의 사항
    1. 본문은 콘텐츠 영역으로 제공된 텍스트 또는 정보를 제공하는 표, 그래프 및 텍스트 이미지의 텍스트 내용을 의미
      • 명도의 차이를 두어야 하는 대상 : 웹페이지 콘텐츠(전체)를 심사의 대상으로 정함
      • 장식적인 문구를 제외한 모든 텍스트에 대해서 명도 대비를 줌
    2. 텍스트 크기가 구분되지 않는 경우나 이미지 텍스트의 경우는 글 모양과 상관없이 windows에서 14pt는 18.66px이고 18pt는 24px로 적용하며, 굵음의 여부와 상관 없이 14pt 이상은 3:1이상, 미만은 4.5:1이상의 명도 대비를 제공하는 경우 준수하는 것으로 인정
      • MacOS는 pt와 px가 같음
      • 운영체제에 따라서 값이 달라지므로 Windows에서는 18.66px, Mac에서는 14px이 14p에 해당
    3. 웹페이지 콘텐츠에 장식 목적으로만 사용한 텍스트, 로고 또는 상호와 같은 텍스트 이미지, 마우스나 키보드를 활용하여 초점을 받았을 때 색이나 명도 대비가 변화하는 콘텐츠, 사용할 수 없음을 표시하기 위하여 명도 대비를 낮춘 회색의 컨트롤이나 입력 서식 등은 이 검사 항목의 적용을 받지 않음
      • 장식 목적으로 사용한 텍스트나 로고 등에 대한 이미지 텍스트와 비활성화를 나타내기 위해 명도를 낮추어 회색으로 적용하는 컨트롤이나 입력 서식은 명도 대비 검사 대상에서 제외
    4. 색상테마 등을 이용하여 전체 웹 사이트의 색상정보를 변경하여 준수한 경우 인정

    2. 자동적으로 재생되는 배경음의 지속시간이 3초 이상, 제어 수단이 페이지의 가장 첫 부분에 제공되는 경우 준수한 것으로 인정

    • 시각 및 상지 장애인들이 마우스 사용자와 같이 원하는 콘텐츠로 쉽게 접근할 수 있게 하기 위함

10. 접근성의 미래

1. 웹 접근성의 올바른 발전 방향

  • 웹 접근성의 문제점
    • 마크업 기술 중심
    • HTML과 CSS 중심의 굴레를 벗어나지 못함
  • 웹 접근성 국제 표준
    • WCAG 1.0 - 2008년에 신기술을 고려, HTML 중심
    • WCAG 2.0 - WCAG 1.0+다양한 신기술 고려
    • KWCAG 2.1 - 동적인 문서나 애플리케이션 영역에 대한 접근성 영역까지 확장
  • 웹 접근성의 올바른 발전 방향
    • 웹 접근성은 잠시의 유행으로 끝나는 것이 아니라 전 세계적인 흐름이자 반드시 필요한 영역
    • 웹 접근성이 마크업 영역에 머무르지 않고, 신기술과 새로운 영역까지 발전
    • 웹 접근성이 신기술을 규제하는 것이 아니라 연구를 통한 접근성 적용이 가능

2. 새로운 기술의 방향

  • HTML5와 ARIA 등 새로운 기술은 웹 접근성을 향상하는 방향으로 발전
  • 화면 낭독기 중심으로 적용하는 접근성 구현 방식이 프로그래밍을 통해 해결
  • 접근성은 UX와 함께 연계하여 발전할 때 공감할 수 있는 방향으로 전환

3. 모바일 접근성

  • 한국형 웹 콘텐츠 접근성 지침 2.1 국가 표준으로 개정
  • 장애인 스마트워크 접근성 가이드북 - 이동이 불편한 장애인들을 위한 업무시설
  • 모바일 서비스 - 실제 장애인들을 위한 접근성은 지침 뿐만 아니라 실생활에서 필요한 정보 즉각적으로 알려줄 수 있을때 발휘
  • 국내의 모바일 애플리케이션 접근성 지침은 7개의 준수항목과 8개의 권고항목으로 구성
  • 모바일 애플리케이션 접근성 지침의 업데이트가 필요
  • 한국형 웹 콘텐츠 접근성 지침 2.0과 비교했을 때 콘텐츠의 이해를 높일 수 있는 항목은 포함하여 적용
  • 모바일 웹 역시 한국형 웹 콘텐츠 접근성 지침 2.0에서 인지되지 못한 접근성 문제가 있으므로 새로운 지침을 제정하거나 타 지침에 추가하여 보안

  • 이에 반해 신체기능이 저하된 장애인과 고령자 등에 대한 접근성은 취약
  • 국내 기업들은 소비자층을 확대하기 위한 새로운 시장 창출, 국제 경쟁력 강화를 목적으로 제품접근성에 관심을 보이기 시작

1.화면 낭독기의 이해 및 사용법

  • 화면 낭독기, 스크린 리더와 같은 명칭의 프로그램은 윈도우 컨트롤, 응용 프로그램, 웹에 제공된 정보를 시각 장애인인 귀로 들을 수 있도록 음성으로 전환
  • 스크린 리더는 웹페이지에 없는 정보를 스스로 판단해서 음성으로 생산하는 것이 아니라 웹 개발자에 의해 제공된 정보만 음성 정보로 제공
  • 가상커서는 스크린 리더에서 인터넷 브라우저를 지원하기 위해 사용하는 개념으로 화면에 표시되지 않은 가상의 UI
  • 스크린 리더는 제품마다 최신기술 지원 여부와 동작방식에 일부 차이가 있으므로 2개 정도의 제품으로 중복 테스트를 하는 것을 권장

13. 센스리더 설치와 활용

1. 이미지의 대체 텍스트

  • 센스리더로 롱데스크 속성에 연결된 설명파일을 보기위해서는 alt enter를 누르면 됩니다.
  • alt값이 정의되지 않은 이미지 - 파일명을 그대로 읽어줍니다.
  • alt값이 제공된 이미지 - alt값을 읽어줍니다.
  • alt값과 title값이 제공된 이미지 - title값을 음성 출력하지 않습니다.
  • alt값에 공백 처리를 한 이미지 - 음성출력하지 않고 넘어갑니다. 이미지를 숨기는 효과
  • longdesc로 이미지에 대한 긴 설명글을 달아논 이미지 - 연결된 긴 설명파일을 볼 수 있습니다.

2. 링크와 title, target

  • tarket="_blank"속성 사용 - 링크텍스트와 새창 링크라고 음성 출력합니다.
  • title="새창열림" 속성 사용 - 링크텍스트와 새창 열림링크 라고 음성 출력합니다.
  • tarket="_blank" title=" 새창열림" 모두 사용 - 링크텍스트와 새창열림, 새창 링크 라고 음성출력합니다. 한가지 속성만 사용하는 것이 좋습니다.
  • tarket="_blank" 링크안에 이미지로 링크텍스트를 제공한 경우 - 링크텍스트와 새창 그래픽 링크라고 음성출력 합니다.

3. 입력 서식의 레이블 제공

  • 별도로 라벨을 제공하지 않은 경우 - 체크박스와 텍스트의 음성이 따로 출력됩니다.
  • title 정보를 제공한 경우 - 화면에는 표시되지 않지만 음성 출력이 됩니다.
  • 라벨로 for, id로 연결해준 경우 - 체크박스와 텍스트가 따로 음성 출력됩니다.
  • 라벨을 for, id로 묶지 않고 라벨로 감싼 경우 - 체크상자와 1:1로 매칭이 되지 않습니다.
  • 라벨 정보와 title 정보를 동시에 제공한 경우 - title정보는 생략되고 라벨 정보만 음성 출력됩니다.
  • 라벨로 사용되는 것이 텍스트가 아니라 이미지로 사용한 경우 - 체크상자와 1:1로 매칭이 되지 않습니다.

4. 페이지, 프레임, 콘텐츠 블록의 제목 제공

  • th abbr 속성 사용 - 긴텍스트를 다 출력하지 않고 길이를 줄여주고 간단하게 음성출력하는 효과를 줄 수 있습니다.

5. 목록 태그

  • 음성출력하기 전에 목록 시작하고 목록 개수를 음성출력 합니다.
  • ul목록 : 앞의 기호가 상황에 따라 음성출력이 되게 되어있습니다.
  • ol목록 : ol의 경우는 앞에 숫자 1,2가 되어 있을 때 그대로 음성출력을 하며 리스트 스타일을 눈으로 처리하게 되면 숫자가 표시 되지 않기 때문에 숫자를 음성출력하지 않습니다.
  • dl목록 : dt와 dd로 구분해서 음성출력하지 않고, 하나의 일반적인 텍스트 같이 음성출력을 합니다.
  • 복잡한 계층 구조는 사용을 피하시는게 실제 시각장애인에게 더 도움을 줄 수 있는 방법입니다.
  • ul을 여러번 사용하는 것 보다는 li를 활용하는 것이 더 효과적인 방법입니다.

6. 테이블의 활용

  • 아랫방향키를 누르다가 테이블 내부로 진입했을 때 ctrl alt 키를 누른 상태에서 방향키를 화면에 보이는 방향대로 우측셀을 읽고자 한다면 crtl alt 아래방향키를 눌러가면서 내용을 음성으로 듣습니다.
  • ctrl alt 위, 아래 방향키를 눌러서 숫자 부분만 이동을 연결해서 읽어줍니다.
  • ctrl alt 좌, 우 방향키를 눌러서 상단에 있는 제목영역까지 함께 연결해서 음성출력합니다.
  • th에서와는 달리 이동하는 방향 셀에 진입하는 방향에 관계 없이 한번에 읽어줍니다.
  • 시각장애인 사용자가, 센스리더 사용자가 테이블의 구조를 이해 하는데 크게 도움이 됩니다.

정지기능 제공의 준수기준

정지기능 제공의 검사내용

3. 깜빡임과 번쩍임 사용 제한

깜빡임과 번쩍임 사용 제한의 이해

  • 검사항목 13. 깜빡임과 번쩍임 사용 제한

깜빡임과 번쩍임 사용 제한의 준수 기준

  1. 초당 3~50회 주기로 깜빡이거나 번쩍이는 콘텐츠를 제공하지 않은 경우
  2. 깜빡임이 3초 미만인 경우

  • 사전 경고없이, 초당 3~50회 깜빡이는 콘텐츠가 존재할 경우
  • 동영상 콘텐츠도 검사 대상에 포함됩니다.

깜빡임과 번쩍임 사용 제한의 검사방법

반복영역 건너뛰기

반복영역 건너뛰기의 이해

  • 검사항목 14. 반복영역 건너뛰기 : 콘텐츠의 반복되는 영역은 건너뛸 수 있어야 함
  • 일반인은 화면 내 원하는 정보를 마우스 동작을 통하여 쉽게 실행할 수 있습니다.
  • 시각장애,운동장애 사용자는 원하는 콘텐츠에 접근하기 위해 탭키를 수십,수백번 눌러야 하는 경우도 있습니다.
  • 페이지마다 반복되는 콘텐츠는 건너뛰기 링크를 제공합니다.

반복영역 건너뛰기의 준수기준

  • 반복영역의 건너뛰기 링크를 제공한 경우 준수한 것으로 인정합니다.

반복영역 건너뛰기 오류 유형

  1. 건너뛰기를 제공하지 않은 경우
  2. 건너뛰기 링크는 제공하고 있으나 동작이 안 되는 경우

반복영역 건너뛰기의 검사 내용

  • 건너뛰기 링크를 제공하였는지 확인
  • 본문으로 이동할 수 있는 기능이 동작하는지 확인
  • 너무 많은 바로가기는 제공하지 않았는지 확인
  • 화면에 보이도록 제공되었는지 확인

반복영역 건너뛰기의 검사방법

  1. 웹 브라우저에서 제공하는 CSS 제거 기능을 통해 CSS스타일을 제거
  2. CSS제거 후 표기되는 화면에서 건너뛰기 링크 확인
    : 로고, 검색창, 서비스 메뉴 등 페이지마다 반복되는 영역을 건너뛰고 주요 콘텐츠로 바로 이동하는 링크를 페이지 시작부분에 위치해야 합니다.
    : 건너뛰기 링크의 존재 여부
    : 건너뛰기 링크가 페이지 최상단 위치 여부
    : 건너뛰기 링크의 총 개수
  3. 건너뛰기 링크가 정상적으로 작동하는지 링크를 선택하여 이동되는지 확인
  4. 본문 바로가기 링크가 화면에 보이도록 제공되었는지 페이지 내에서 확인
    : 화면 내에 본문 바로가기 링크를 항상 표시되도록 하며, 탭 이동시 화면에 표시되는 지 확인

반복영역 건너뛰기의 주의 사항

  1. 반복되는 영역을 건너뛰고 본문으로 바로 갈 수 있는 기능의 제공
  2. <body>요소 다음으로 제공
  3. display:none;속성과 같이 CSS를 통해 화면에서 숨기는 경우, CSS를 제거한 상태에서 키보드 접근도 가능하고 동장에도 문제가 없지만, CSS가 적용된 상태에서는 시각적으로 표시도 되지 않음.
  4. 스크린리더 뿐만 아니라 일반적인 키보드 접근이 불가함
  5. 건너뛰기 링크가 필요없는 경우지만 제공된 경우, 불필요한 건너뛰기 링크라도 제공된 건너뛰기 기능이 동작되지 않으면 감점

5. 제목 제공

제목 제공의 이해

  • 검사항목 15. 제목 제공 : 페이지, 프레임, 콘텐츠 블록에는 적절한 제목을 제공해야 함

제목 제공 준수 기준

  1. 페이지, 프레임, 콘텐츠 블록에 적절한 제목을 제공한 경우 준수한 것으로 인정합니다.
  2. 페이지 제목에 title속성 미 제공, 내용과 다른 의미의 제목을 사용한 경우
  3. 페이지 제목에 반복되는 특수문자를 제공한 경우
  4. 페이지 제목의 분류가 더 가능하나, 상위 범주로 제목을 제공한 경우
  5. frame, iframe, frameset 요소의 title속성이 없거나, 속성 값을 비워 둔 경우, 내용이 부적절한 경우
  6. 내용 또는 기능이 없는 프레임에 title을 비워두거나, 제공하지 않은 경우
  7. 콘텐츠 블록에 h1~6을 사용하여 제목을 제공하지 않은 경우

제목 제공의 검사 내용

  • 적절한 제목을 제공하였는지 확인합니다.
  • 프레임마다 각 프레임을 설명하는 간단명료한 제목을 제공하였는지 확인합니다.
  • 콘텐츠 블록에 h1~6요소를 이용하여 적절한 제목을 제공하였는지 확인합니다.

제목 제공 검사방법

  1. 웹 페이지 제목 확인
    사용자가 가장 먼저 인식하는 정보로, 화면낭독기나 점자정보 단말기가 가장 먼저 접하는 정보입니다.
    내용을 제목으로 간단명료하게 지정하면 원하는 페이지를 쉽게 찾을 수 있습니다.
    제목만으로도 웹페이지의 전체를 파악하는 수단으로 사용됩니다.
    웹브라우저의 상단제목이나 탭제목으로 확인하여 페이지의 성격과 콘텐츠의 내용이 잘 드러나 있는지 확인합니다.
    미준수 예시 : 사이트 명 등을 동일 제목으로 제공시, 제목을 꾸미기 위해 특수문자를 중복 사용
  2. 프레임 제목 확인
    외부 페이지의 콘텐츠를 쉽게 파악할 수 있는 내용
    페이지 소스를 확인하여 frame, iframe, frameset을 검색하여 title속성을 이용하여 적절한 제목을 제공했는지 확인합니다.
  3. 콘텐츠 블록 확인
    h1~6를 사용하여 적절히 제공하여 콘텐츠에 쉽게 접근, 내용파악에 용이합니다.

제목 제공의 주의 사항

  1. 프레임 제목으로 포함된 콘텐츠를 유추할 수 있는 간결한 제목 제공
  2. 게시판의 '목록, 읽기, 쓰기' 페이지의 제목을 목록, 읽기, 쓰기의 상태를 페이지 제목을 통해 구분할 수 있도록 제공
  3. 여러 단계로 구성되어 있는 페이지의 경우, 페이지 제목만으로도 각 단계를 인지할 수 있도록 제공

2. 사용자 요구에 따른 실행

검사항목 18. 사용자 요구에 따른 실행

  1. 준수 기준
    • 사용자가 의도하지 않은 기능이 자동 실행되지 않도록 제공한 경우 준수한 것으로 인정합니다.
    • 링크나 버튼 등으로 연결된 콘텐츠가 새 창으로 제공될 경우 사전에 안내해야 합니다.

    • 새 창 및 팝업이 경고없이 열리는 지 확인

  2. CSS 제거된 화면에서 콘텐츠 순서가 논리적으로 구성되어있는지 확인합니다. 상에서 하, 좌에서 우로
  3. 탭 메뉴가 논리적으로 구성되었는지 확인합니다.
  4. 메뉴의 계층 표현을 확인합니다.

표의 구성

  • 제목 또는 요약정보를 <caption>요소와 summary속성으로 제공하였는지 확인
  • 복잡한 표를 제공시 headers 속성으로 <td>에서 <th>의 id를 참조 또는 scope속성으로 <th>요소에 <td>요소의 범위를 지정하여 제공하였는지 확인
  • 배치용 테이블 <caption>요소, summary속성, <th>요소를 이용하지는 않았는지 확인

18. 검사항목(19,20,21,22) 평가 방법

1. 레이블 제공

검사항목21. 레이블 제공

  • 입력서식에는 반드시 레이블을 제공하여 입력창에 대한 설명을 제공해야 합니다.
  • 레이블이 제공된 입력 서식은 화면을 볼 수 없어도 화면 낭독기에서 입력서식에 어떠한 값을 입력해야 하는지 알 수 있게 됩니다.

검사 개요

  1. 준수 기준
    • 입력 서식에 대응하는 레이블을 제공한 경우 준수한 것으로 인정합니다.
  2. 오류 유형
    • 19-1 : <input type="image, hidden, submit, button, reset">을 제외한 모든 input, textarea, select 요소에 1:1 대응하는 <label>요소 또는 title속성을 제공하지 않은 경우
    • 19-2 : <input>의 id와 <lable>의 for가 다르거나, 페이지 안에 같은 id가 있는 경우
    • 19-3 : <select> 요소의 첫 번째 <option>이 레이블 역할을 대신하는 경우
  3. 검사 내용
    • 입력서식에 <label>요소나 title 속성을 이용하여 레이블을 제공하였는지 확인합니다.
    • <lable>요소를 이용하여 레이블을 제공한 경우, label for="값"과 입력서식의 id="값"이 적절히 짝지어졌는지 확인합니다.
  4. 레이블을 제공해야 하는 대표적인 유형
    • 편집창 : type=text
    • 암호 편집창 : type=password
    • 다중 편집창 : textarea
    • 라디오 버튼 : type=radio
    • 체크박스 : type=checkbox
    • 콤보박스 : select
    • 파일 선택창 : type=file

검사 방법

  1. 페이지 소스에서 레이블 제공 여부 검사
    • 입력서식을 찾아 '요소 검사'를 클릭합니다.
    • 레이블 요소나 title속성을 이용하여 레이블을 제공하였는지 확인합니다.
  2. label 요소의 for와 id값 비교
    • label요소의 for값과 input요소의 id값을 비교하여 두 값이 같다면 준수, 같지 않다면 미준수로 평가합니다.
  3. 개발자 도구의 요소 선택자를 이용하여 검사
    • display:none과 같이 숨김처리 한 경우 시각적으로 보이지 않고 보조기기에서도 컨텐츠를 없는 것 처럼 처리합니다.
    • 보조기기에서 인지할 수 없게 마크업 처리 되었다면 미준수로 평가합니다.
  4. openwax를 이용하는 방법
    • openwax를 실행하고 '현재 페이지 검사'를 클릭합니다.

2. 오류 정정

검사항목 22. 오류 정정

  • 입력 서식 작성시 항목이 누락되었거나 잘못 입력할 때 사용자에게 오류 안내를 하고 수정이 필요한 곳으로 포커스 이동을 하여 사용자가 바로 수정할 수 있게 해야 합니다.

검사 개요

  1. 준수 기준
    • 입력 서식에 대응하는 레이블을 제공한 경우 준수한 것으로 인정합니다.
  2. 오류 유형
    • 22-1 : 입력 서식을 잘못 작성한 경우 해당 서식 필드로 초점을 이동하도록 제공하지 않거나 해당 서식의 전송버튼을 눌렀을 때, 입력 내용이 모두 사라지는 경우
    • 22-2 : 오류 발생 시, 정정할 수 있는 수단을 제공하지 않는 경우
    • 22-3 : 입력 정정방식 또는 내용을 잘못 제공한 경우
  3. 검사 내용
    • 입력 값을 잘못 기재하였을 경우, 오류 내용과 수정방법을 제공하였는지 확인하고 다시 입력할 수 있도록 오류 입력 서식으로 키보드 포커스가 이동하는지 확인합니다.
    • 오류 내용 확인 후 기재하였던 입력 내용이 삭제 되는 지 확인합니다.

검사 방법

  1. 오류 메시지 검사
    • 오류에 대한 정보를 제공하고 정정할 수 있는 기능을 제공하였는지 확인합니다.
    • 오류 메시지가 잘못된 부분은 없는지도 확인합니다.
  2. 오류 발생시 입력 값 초기화 여부 검사
    • 오류 내용 확인 후 기재하였던 입력 내용이 삭제되었는지 확인합니다.
    • 기재하였던 입력 내용이 삭제되었다면 미준수로 평가합니다.
    • 단, 인터넷 뱅킹처럼 보안상의 문제로 초기화하는 경우는 예외사항으로 봅니다.
  3. 오류 발생시 포커스 이동 검사
    • 오류 내용 확인 후 오류가 난 입력서식으로 포커스가 이동되는지 확인합니다.

3. 마크업 오류 방지

검사항목 23. 마크업 오류 방지

  • 열고 닫음, 중첩 관계 및 속성 선언에 오류가 없어야 합니다.

검사 개요

  1. 준수 기준
    • 마크업 언어 요소의 열고 닫음, 중첩 관계 및 속성 선언에 오류 없이 제공한 경우 준수한 것으로 인정합니다.
  2. 오류 유형
    • 21-1 : 태그의 열고 닫음 오류
    • 21-2 : 태그의 중첩 오류
    • 21-3 : 중복 선언된 속성 오류1,2
  3. 검사 내용
    • 해당 마크업 언어의 문법을 최대한 준수하여 제공하였는지 확인합니다.
    • 열고 닫음, 중첩관계, 속성 선언에 대한 오류는 없는지 확인합니다.

검사 방법

  1. W3C의 Markup Validation Service를 통해 확인
    • http://validator.w3.org/로 접속 후 Address 편집창에 웹 주소 기재 후 Check
    • 마크업 문법 준수 결과 확인 후 Error가 있는 경우, Validation Output의 내용을 확인
  2. openwax를 이용하는 방법
    • openwax를 이용하여 '현재 페이지 검사'를 클릭합니다.
    • 마크업 오류 방지 항목을 확인합니다.

4. 웹 애플리케이션 접근성 준수

검사항목 24. 웹 애플리케이션 접근성 준수

  • 콘텐츠에 포함된 웹 애플리케이션은 접근성이 있어야 합니다.

검사 개요

  1. 준수 기준
    • 웹 애플리케이션의 자체 접근성을 준수하여 제공한 경우 준수한 것으로 인정합니다.
  2. 오류 유형
    • 24-1 : 웹 애플리케이션이 자체적인 접근성이 없으며 사용자가 선택할 수 있는 대체 콘텐츠가 존재하지 않거나 대체 콘텐츠를 제공하더라도 핵심기능을 동등하게 제공하지 못한 경우

검사 방법

  1. 웹 페이지에서 마우스 오른쪽 클릭 시 나오는 팝업 메뉴 확인
  2. 부가 애플리케이션의 기능을 찾아 자체적인 접근성 제공 여부 확인
    • 이미지에 대한 대체 텍스트 제공 여부
    • 키보드에 대한 논리적 이동 및 초점 표시 확인
    • 키보드로 기능을 사용할 수 있는 지 확인
    • 보조기기(스크린리더 등)와의 지원 여부
  3. 동등한 대체 콘텐츠 제공 여부 확인
  4. 접근성 검사 도구 - UIA(User Interface Automation) Verify 사용
  5. 주의 사항
    • 자바 스크립트를 지원하지 않는 환경에서는 평가하지 않음
    • 직접적으로 접근성을 구현하기 어려운 경우 대체 수단을 제공해서 우회할 수 있는 경로를 제공하여야 함 (단, 접근성 구현과 동등한 수준으로 제공)
    • 플래시 버전은 그대로 두고, 플래시를 이용하지 못하는 사용자에게 일반 HTML로 제공된 대체 수단을 제공하여 이용할 수 있다면 준수로 인정
